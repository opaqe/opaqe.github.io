# 포스트 작성 규칙 (spring/_posts)

`spring/_posts/`에 글을 쓸 때 따르는 규칙. 기존 글
(`2026-06-19-m0-1-spring-boot-scaffold.md`, `m0-2`, `m0-3`)에서 추출한 컨벤션이다.
새 글은 이 문서를 기준으로 작성하고, 어긋나는 기존 글은 여기에 맞춰 고친다.

---

## 1. 파일명

| 규칙 | 형식 | 예 |
|---|---|---|
| 패턴 | `YYYY-MM-DD-<slug>.md` (Jekyll 표준) | `2026-06-19-m0-3-postgresql-flyway.md` |
| 날짜 | 글의 `date`와 같은 날짜 | `2026-06-19` |
| slug | 소문자 + 하이픈(kebab-case). 마일스톤 번호 + 핵심 키워드 | `m0-3-postgresql-flyway` |
| 마일스톤 표기 | `m{대}-{소}` 형태를 slug 맨 앞에 | `m0-1`, `m0-2`, `m0-4` |

> 이 가이드 파일처럼 **포스트가 아닌 문서**는 `_posts/` 바깥에 둔다.
> `_posts/` 안의 파일은 날짜 파일명 규칙을 지켜야 Jekyll이 포스트로 인식한다.

---

## 2. Front matter (필수)

모든 포스트는 `---`로 감싼 YAML front matter로 시작한다. **5개 키 모두 필수.**

```yaml
---
layout: post
title: "M0-3: PostgreSQL + Flyway 연동 (Spring Boot 3.5 / Java 17)"
date: 2026-06-19 17:00:00 +0900
tags: [spring-boot, postgresql, flyway, testcontainers, jpa]
description: "앱을 PostgreSQL에 연결하고 Flyway로 스키마 마이그레이션을 자동 적용, Testcontainers로 H2 없이 운영과 동일한 DB로 통합 테스트를 그린으로 만든 과정."
---
```

### 2-1. 키별 규칙

| 키 | 값 | 규칙 |
|---|---|---|
| `layout` | `post` | 항상 `post` 고정 (따옴표 없음) |
| `title` | `"M0-3: 제목 ..."` | **큰따옴표로 감쌈**. `M{마일스톤}: ` 접두 + 콜론 뒤 제목. 보조 설명은 괄호로 `(Spring Boot 3.5 / Java 17)` |
| `date` | `2026-06-19 17:00:00 +0900` | `YYYY-MM-DD HH:MM:SS +0900` (KST). 따옴표 없음. 시간까지 명시 |
| `tags` | `[spring-boot, postgresql, ...]` | 인라인 배열. 소문자 kebab-case. 핵심 기술 키워드 |
| `description` | `"한 줄 요약"` | **큰따옴표로 감쌈**. 글의 목표/결과를 1문장으로. SEO·목록 노출용 |

### 2-2. `date` 시간 순서

같은 날 여러 글이면 시:분:초로 순서를 구분한다.
(예: M0-1 `15:00:00` → M0-2 `16:01:00` → M0-3 `17:00:00`)

### 2-3. `tags` 운용

- 글에서 실제로 다룬 기술만 넣는다.
- 공통 베이스 태그(`spring-boot`)를 먼저, 그다음 그 글 고유 키워드.
- 예: M0-1/M0-2 `[spring-boot, gradle, java, actuator]`,
  M0-3 `[spring-boot, postgresql, flyway, testcontainers, jpa]`

## 3. 본문 도입부

front matter 닫는 `---` 다음 한 줄 띄우고, 글의 목표를 **한 문단 또는 인용블록(`>`)**으로 먼저 제시한다.

```markdown
> 앱을 실제 DB에 붙이고, 스키마 마이그레이션을 자동 적용하고, **H2가 아닌 운영과 동일한 PostgreSQL**로
> 통합 테스트를 그린으로 만드는 단계.
```

또는 인용블록 없이 한 문장:

```markdown
빈 Spring Boot 앱을 세우고 `/actuator/health`가 200을 반환하는 것까지를 목표로, ...
```

핵심 단어는 `**굵게**`로 강조한다.

## 4. 섹션 구조

`## 1. 제목` 형식의 **번호 매긴 H2**로 구성한다. 표준 흐름:

| 순서 | 섹션 예 | 내용 |
|---|---|---|
| 1 | `## 1. 목표` / `## 1. 한눈에 보기` / `## 1. 무엇을 / 왜` | 목표·DoD·범위·스택을 표로 |
| 2 | `## 2. 핵심 개념 정리` / `## 2. 학습 개념 정리` | 새로 익힌 개념을 표로 정의 |
| 3 | `## 3. 의사결정 (정리)` | 선택지·선택·근거를 표로 (탈락안 포함) |
| 4 | `## 4. 구현 / 소스 전체` | 파일 경로별 전체 소스 |
| 5 | `## 5. 테스트` | 테스트 클래스·검증 내용 |
| 6 | `## 6. 실행 / 검증` | 명령어와 기대 결과 |
| 7 | `## 7. 트러블슈팅` / `## 5. 겪은 문제와 해결` | 증상·원인·해결 3열 표 |
| 8 | `## DoD (완료 기준)` | 체크 항목 표 |
| 9 | `## 핵심 요약` / `## 핵심 정리` | 키워드·한 줄 표로 마무리 |
| 10 | `## Git 작업 흐름`, `## 다음 단계` | (선택) 브랜치/커밋/PR, 다음 마일스톤 예고 |

> 모든 섹션을 다 쓸 필요는 없다. **목표 → 개념/의사결정 → 구현 → 검증 → 요약** 골격은 유지한다.
> 섹션 번호는 글 안에서 1부터 연속으로 매긴다. 하위 구분은 `### 3-1.`처럼 표기.

---

## 5. 표 (이 블로그의 핵심 표현 수단)

내용 대부분을 **표로 정리**한다. 산문 나열보다 표를 우선한다.

- 의사결정: `| 결정 | 선택 | 근거 |` 또는 `| 쟁점 | 선택 | 탈락안 / 근거 |`
- 개념 정리: `| 개념 | 설명 |` / `| 개념 | 한 줄 정의 | 다루는 대상 | 동작 시점 |`
- 트러블슈팅: `| 증상 | 원인 | 해결 |`
- 요약: `| 키워드 | 한 줄 |` / `| 포인트 | 내용 |`
- 검증 결과: `| 항목 | 결과 |`

---

## 6. 코드 블록

- 소스 앞에 **파일 경로를 굵게** 적는다: **`backend/build.gradle`** (헤딩 `###`로 써도 됨).
- 언어 펜스를 정확히: ```` ```groovy ```` ```` ```java ```` ```` ```yaml ```` ```` ```sql ```` ```` ```bash ```` ```` ```powershell ```` 등.
- 변하지 않는 보일러플레이트보다 **그 단계에서 실제 바뀐/추가된 부분**을 보이고, 주석으로 이유를 단다.
  (예: `// 미사용(부팅 확인용)`, `# 환경값 우선, 없으면 8090`)
- Windows/PowerShell 환경 차이가 있으면 별도 펜스로 함께 제시한다.

## 7. 강조 / 인용

| 표기 | 용도 |
|---|---|
| `**굵게**` | 핵심 키워드, 원칙, 주의점 |
| `` `코드` `` | 파일명·설정키·명령·클래스명 등 식별자 |
| `> 인용블록` | 목표 제시, 보충 설명, 주의(callout) |
| ✅ / ❌ / ☑ | 필요/불필요, 체크 항목 표시 |

---

## 8. 문체

- **한국어**, ~다/~한다 체(평서·개조식). 기술 식별자는 영문 그대로.
- 1인칭/대화체 대신 **정리·기록 톤**. "왜 이 선택을 했는가"의 근거를 항상 남긴다.
- 실패·트러블슈팅을 숨기지 않고 **증상–원인–해결**로 기록한다.

---

## 9. 체크리스트 (글 올리기 전)

- [ ] 파일명이 `YYYY-MM-DD-<slug>.md` 형식인가
- [ ] front matter 5개 키(`layout`, `title`, `date`, `tags`, `description`)가 모두 있는가
- [ ] `title`·`description`이 큰따옴표로 감싸였는가
- [ ] `date`가 `... +0900`까지 포함하고, 같은 날 글이면 시간 순서가 맞는가
- [ ] `tags`가 소문자 kebab-case 인라인 배열인가
- [ ] 도입부에 목표 문장/인용블록이 있는가
- [ ] 섹션이 번호 매긴 H2이고 목표→구현→검증→요약 골격을 갖췄는가
- [ ] 소스마다 파일 경로가 명시됐는가
- [ ] 의사결정·트러블슈팅이 표로 정리됐는가
